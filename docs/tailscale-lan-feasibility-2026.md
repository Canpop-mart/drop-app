# Tailscale virtual-LAN for Drop - feasibility research (2026-06)

## Executive summary

Embedding Tailscale as a zero-config virtual-LAN inside Drop is technically feasible via Tailscale's embeddable tsnet/libtailscale (a Go-via-cgo C library that runs a full Tailscale node in userspace — no TUN device, no daemon, no root), and the API directly supports the server-mints-auth-key zero-login flow Drop wants (Server.AuthKey/TS_AUTHKEY, Ephemeral nodes, ACL tags, custom ControlURL for self-hosted Headscale). However, the goal hits four concrete friction points: (1) build complexity — embedding pulls a Go toolchain + cgo into every-OS CI, and the older libtailscale spins up a full Go runtime in-process (a known crash source when mixed with other runtimes), while the cleaner pure-Rust replacement (tailscale-rs) is explicitly experimental, unaudited, DERP-relay-only with no P2P/NAT-traversal and no DNS, so it is not yet usable; (2) userspace mode gives the host process a tailnet IP but NOT a transparent virtual interface — an external game .exe that is not proxy-/tsnet-aware cannot transparently route over the overlay, which is the single biggest architectural obstacle to "any game just works"; (3) NAT traversal succeeds >90% in typical conditions but falls back to slower/higher-latency DERP relays when both peers are behind hard/symmetric NAT, degrading real-time game latency for some user pairs; (4) the layer-3 overlay does not forward UDP broadcast/multicast, so LAN peer discovery needs per-game glue (e.g. writing peer overlay IPs into Goldberg config). Working real-world embeds exist (SockTail uses tsnet.Server in a single binary with no daemon), and Drop already maintains its own libtailscale fork — but the cross-platform embedded-node build is closer to "weeks" than "a weekend" once the Go/cgo CI, the userspace-routing-for-external-games problem, and per-game discovery glue are all accounted for. The research did NOT directly verify the three crux questions the parent asked about most (whether Goldberg custom_broadcasts.txt actually works over Tailscale, free-tier/Headscale auth-flow limits, and the ZeroTier/libzt L2 comparison) — these remain open.

## Findings

### 1. [high] (vote 3-0 (one supporting sub-claim 2-1 on a pedantic 'OS network stack' wording nuance))
libtailscale/tsnet embeds a complete Tailscale node directly into the host process in pure userspace (gVisor TCP/IP stack) — giving the process a tailnet IP with no TUN device, no separate tailscaled daemon, and no root/admin privileges. This is the correct technical primitive for embedding a Tailscale node inside Drop's Tauri client.

**Evidence:** Four unanimous (3-0) primary-sourced claims converge: libtailscale is 'a C library that embeds Tailscale into a process... get an IP address on a tailnet, entirely from userspace' (tailscale.dev blog + official repo); tsnet 'runs a fully self-contained Tailscale node inside your process using a userspace TCP/IP stack (gVisor)' with 'No root privileges required. No system daemons to install or manage' (pkg.go.dev godoc, verified against tailscale.go source calling netstack.Create()). Userspace mode is the SAME constraint Tailscale built for container platforms (Heroku/Cloud Run) that lack /dev/net/tun. One pedantic caveat surfaced in a 2-1 vote: the node still opens an ordinary unprivileged UDP socket to send WireGuard packets, so 'does not touch the OS network stack' is loose — it touches the unprivileged socket layer but no privileged TUN/driver/admin layer.

**Sources:** https://tailscale.dev/blog/libtailscale  |  https://github.com/tailscale/libtailscale  |  https://pkg.go.dev/tailscale.com/tsnet  |  https://tailscale.com/docs/features/tsnet  |  https://tailscale.com/kb/1112/userspace-networking  |  https://tailscale.com/kb/1177/kernel-vs-userspace-routers

### 2. [high] (vote 3-0)
Embedding tsnet/libtailscale forces a Go toolchain + cgo into Drop's build pipeline on every OS. libtailscale is built via `go build -buildmode=c-archive` (static libtailscale.a) or `-buildmode=c-shared` (.so/.dll/.dylib), requiring CGO_ENABLED=1 and Go (lib.rs lists Go v1.20+, but upstream go.mod is now Go 1.25.x). This is the dominant build-complexity cost for cross-platform CI.

**Evidence:** Unanimous primary-sourced claims: the official libtailscale README and tailscale.dev blog both document `-buildmode=c-archive` (libtailscale.a + tailscale.h) and `-buildmode=c-shared`, supporting both static and dynamic linking; these modes require CGO_ENABLED=1 and a Go toolchain by definition. The Rust `tsnet` crate's own Requirements list 'Go v1.20 or higher' (lib.rs), confirming Go/cgo is pulled in. The Go-version floor has drifted upward (upstream go.mod declares go 1.25.5; passcod fork says 1.21+), so the 1.20 figure is conservative/stale but still true. Note: Drop maintains its own fork at github.com/Drop-OSS/libtailscale, reinforcing that the real build pipeline needs the Go toolchain.

**Sources:** https://github.com/tailscale/libtailscale  |  https://tailscale.dev/blog/libtailscale  |  https://lib.rs/crates/tsnet  |  https://pkg.go.dev/cmd/go/internal/help  |  https://github.com/Drop-OSS/libtailscale

### 3. [high] (vote 3-0)
The older libtailscale spins up a full Go runtime inside the host process on first call; this is a documented crash source when mixed with other language runtimes (the two fight over signal handlers and process lifecycle). This is a real risk for embedding inside a complex Tauri/Rust process and was the explicit motivation for Tailscale building tailscale-rs.

**Evidence:** Tailscale's own engineering blog states verbatim 'the first time you call into libtailscale, it spins up an entire Go runtime inside your process' and that 'mixing libtailscale with a Ruby VM is a quick way to end up with a crash, because the two runtimes will step on each other's toes.' Independently corroborated by Go core-team sources (golang/go issues #65050 panic from multiple c-shared libraries, #59724 heap corruption with cgo on Windows; golang-nuts signal-collision threads; konradreiche.com 'A cautionary tale of cgo embedding'). The Ruby example generalizes to other runtimes per the blog's general statement plus these Go-core sources. Note: a pure Rust/Tauri host has no competing VM the way Ruby/Python does, but the Go runtime taking over process lifecycle remains a stability concern.

**Sources:** https://tailscale.com/blog/tailscale-rs-rust-tsnet-library-preview  |  https://github.com/golang/go/issues/65050  |  https://github.com/golang/go/issues/59724

### 4. [high] (vote 3-0)
The cleaner alternative — tailscale-rs, a from-scratch pure-Rust reimplementation of Tailscale's core with NO embedded Go runtime, designed to embed cleanly as a guest in another program — exists but is NOT usable for Drop today: it is explicitly experimental, unstable, insecure, unaudited, has NO peer-to-peer/NAT-traversal (all traffic relays through DERP with limited throughput), and NO DNS resolution (dial by IP only).

**Evidence:** Multiple unanimous primary claims: README states 'This software is unstable and insecure... do not build production software using these libraries or rely on it for data privacy until we... complete a third-party audit', gated behind TS_RS_EXPERIMENT=this_is_unstable_software, 'assume... anything you send or receive could be in the clear on the public Internet.' Tailscale blog (April 15, 2026): 'experimental preview (do not use it in production yet!).' Critically for gaming: 'Peer-to-peer communication and NAT traversal are TODO, so all traffic to other devices goes through DERP and will have limited throughput' and 'There's no DNS resolution, so you have to dial things by IP address.' Latest release v0.3.3 (May 20, 2026) still pre-1.0 with the flag required. A prior claim that 'tailscale-rs already supports TCP/UDP sockets + DERP communication, the core capability for game traffic' was REFUTED 0-3. So tailscale-rs solves the Go-runtime problem but reintroduces DERP-only/no-NAT-traversal, which is worse for real-time game traffic than mature libtailscale.

**Sources:** https://github.com/tailscale/tailscale-rs  |  https://tailscale.com/blog/tailscale-rs-rust-tsnet-library-preview

### 5. [high] (vote 3-0)
The existing Rust `tsnet` crate that wraps Go libtailscale (the lineage of Drop's orphaned crate) is immature and effectively unmaintained: a single unstable 0.1.0 release published March 12, 2023, never updated since, with a failed docs.rs build. Drop would be relying on a stale wrapper or its own fork rather than a maintained dependency.

**Evidence:** Verified across three primary registries: crates.io API returns only version 0.1.0, created 2023-03-12, updated_at identical to created_at (never updated), ~2,164 downloads; lib.rs shows '0.1.0 / Mar 12, 2023 / 1 unstable release'; docs.rs shows only 0.1.0 and 'docs.rs failed to build tsnet-0.1.0.' ~3.25 years with zero updates as of June 2026. The distinct, actively-maintained official `tailscale-rs` crate is a different project (and itself experimental, see prior finding). This matches the parent's context that Drop has an 'orphaned Rust crate wrapping libtailscale' with a latent Windows compile error.

**Sources:** https://lib.rs/crates/tsnet  |  https://crates.io/api/v1/crates/tsnet  |  https://docs.rs/crate/tsnet

### 6. [high] (vote 3-0)
tsnet's Server API directly supplies the client-side primitives for Drop's zero-login, server-mints-auth-key flow: Server.AuthKey (preferred over TS_AUTHKEY env), Server.Ephemeral (ephemeral node registration), Server.AdvertiseTags (ACL tags for tailnet isolation), and Server.ControlURL (custom coordination server URL — i.e. point at self-hosted Headscale). This validates that the non-interactive auth model is achievable at the library level.

**Evidence:** Live-fetched official godoc confirms all four fields verbatim: AuthKey '...preferred over the TS_AUTHKEY environment variable'; Ephemeral 'register as an Ephemeral node'; AdvertiseTags 'ACL tags... referenced from the ACL policy document'; ControlURL 'Custom coordination server URL... defaults to TS_CONTROL_URL.' Pointing the control URL at self-hosted Headscale is corroborated by Tailscale's own custom-control-server docs and the Headscale (juanfont/headscale) project. CAVEATS that bound the zero-fuss claim: (a) these are only the CLIENT primitives — the end-to-end flow still needs a separate server-side key-minting endpoint (outside tsnet's scope); (b) godoc warns 'advertising a tag on the client doesn't guarantee the control server will allow the node to adopt that tag'; (c) GitHub issue #16840 notes ControlURL can be overridden by syspolicy in an edge case.

**Sources:** https://pkg.go.dev/tailscale.com/tsnet  |  https://tailscale.com/docs/how-to/set-up-custom-control-server  |  https://github.com/juanfont/headscale  |  https://github.com/tailscale/tailscale/issues/16840

### 7. [high] (vote 3-0)
In userspace networking mode there is NO transparent virtual network interface — the overlay is reachable only by code that is proxy-aware (SOCKS5/HTTP proxy) or uses the library's own tsnet Listen/Dial. A standalone third-party game .exe that knows neither cannot transparently bind/route over the Tailscale overlay. This is the single biggest architectural obstacle to the 'any game just works' goal.

**Evidence:** Primary Tailscale docs state verbatim that in userspace mode 'tailscaled functions as a SOCKS5 or HTTP proxy which other processes... can connect through' and apps must opt in via ALL_PROXY/HTTP_PROXY env vars — explicitly contrasted with kernel/TUN mode where any app routes transparently. For the tsnet/libtailscale embedding path specifically, traffic goes via the library's Server.Listen/Server.Dial over the embedded gVisor netstack, NOT a SOCKS proxy — so the conclusion is even stronger: an external game binary that calls neither a proxy nor the host app's tsnet Dial gets no transparent overlay interface. Verifier explicitly flagged this as 'a genuine friction point for the zero-fuss LAN-gaming goal: without a real TUN device, an external game .exe cannot transparently bind/route over the Tailscale overlay.' (Mitigation would require running Tailscale in kernel/TUN mode with a real interface — which reintroduces driver install, admin rights, and the zero-config compromise — or per-game userspace shims.)

**Sources:** https://tailscale.com/kb/1112/userspace-networking  |  https://tailscale.com/docs/concepts/userspace-networking  |  https://pkg.go.dev/tailscale.com/tsnet  |  https://tailscale.com/docs/features/tsnet

### 8. [high] (vote 3-0)
Real-time game latency over Tailscale depends on getting a DIRECT P2P connection. NAT traversal succeeds >90% of the time in typical conditions, but when BOTH peers are behind a hard/symmetric NAT, Tailscale cannot establish a direct link and falls back to a DERP relay, which is materially slower and higher-latency (observed 50-280ms+ relayed vs single-digit-ms direct). This degrades game latency for the minority of user pairs behind hard NAT/CGNAT/cellular.

**Evidence:** A cluster of unanimous primary claims: 'Tailscale can't establish a direct connection if both devices are behind a hard NAT'; 'Direct connections usually provide the lowest latency and highest throughput, while relayed connections are a fallback'; 'DERP servers... are generally slower than direct connections and may offer lower maximum throughput'; 'Every connection technically begins via DERP... then upgrades to a direct link if possible'; and 'success rates for direct NAT traversal well north of 90% in typical conditions... more than nine out of 10 connections... end up being direct P2P links.' The >90% is Tailscale's self-reported internal metric ('typical conditions') with no independent third-party measurement; CGNAT/symmetric-NAT/cellular subpopulations relay far more often (GitHub #11112). Newer Peer Relays add a Direct > Peer-Relay > DERP tier but peer relays are still a relay (3-5ms added) and require a configured relay device — extra setup that does not exist by default in a zero-config Drop deployment.

**Sources:** https://tailscale.com/docs/reference/connection-types  |  https://tailscale.com/blog/nat-traversal-improvements-pt-1  |  https://tailscale.com/blog/how-nat-traversal-works

### 9. [high] (vote 2-0)
A real, current, working tsnet-embed reference exists: SockTail embeds a Tailscale node into a single Go binary via tsnet.Server using an auth key, with no external tailscaled daemon, no config files, and no dependencies — demonstrating the exact zero-daemon userspace embed pattern (no TUN device) that Drop would replicate.

**Evidence:** SockTail's repo (Yeeb1/SockTail, pushed 2025-10-03, current) describes 'Tailscale's embedded client (tsnet). Zero config, no daemon, no persistence' and the README confirms it 'creates a tsnet.Server using the provided (or embedded) Tailscale auth key', is 'Fully self-contained, no configuration files', with 'no external dependencies.' The no-TUN/userspace property follows necessarily from using tsnet.Server (per official godoc). Note: this is a 2-0 vote (fewer verifiers) and the 'no TUN device' element is inferred from tsnet's documented behavior rather than stated word-for-word in SockTail's README. A separate claim that SockTail validates the pre-auth-key zero-login model specifically was logged with a 0-0 (no verifier votes recorded), so treat the auth-key-from-server angle as unconfirmed by this source.

**Sources:** https://github.com/Yeeb1/SockTail  |  https://pkg.go.dev/tailscale.com/tsnet

### 10. [medium] (vote synthesis (no single adversarial vote; built from 3-0 verified facts))
Realistic effort + phasing: the cross-platform embedded-node build is closer to 'weeks' than 'a weekend.' The library primitives exist (embed node, mint auth keys, ephemeral/tagged nodes, point at Headscale), but the genuinely hard parts are (a) Go/cgo CI across Windows/Linux/macOS/SteamOS + the latent Windows compile error, (b) routing an EXTERNAL game .exe over a userspace overlay with no transparent interface, and (c) per-game discovery glue (writing peer overlay IPs into configs) since L3 forwards no broadcast/multicast.

**Evidence:** This is a synthesis/judgment finding derived from the verified facts rather than a single quoted source. Straightforward (library-supported): embedding the node, the server-side auth-key minting endpoint, ephemeral/tagged join, ControlURL→Headscale. Hard (verified as friction): cross-OS Go+cgo build complexity and the Go-runtime-in-process crash risk (libtailscale) vs the not-yet-ready tailscale-rs; the no-transparent-interface problem for external game binaries in userspace mode; NAT-traversal DERP fallback for hard-NAT pairs; and L3 not forwarding broadcast/multicast requiring per-game discovery glue. Confidence is medium because the effort estimate is inference, and three of the parent's specific crux questions (Goldberg custom_broadcasts.txt reliability over Tailscale, free-tier/Headscale auth limits, ZeroTier/libzt L2 comparison) were NOT directly verified by any surviving claim.
